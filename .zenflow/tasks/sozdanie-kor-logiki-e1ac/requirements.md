# Product Requirements Document (PRD)
# Платформа для проведения соревнований с экспертной оценкой

## 1. Обзор продукта

### 1.1 Назначение
Веб-система для проведения командных соревнований с экспертной оценкой (идеатоны, хакатоны, кейс-чемпионаты и т.п.). Позволяет организатору управлять мероприятием, назначать задания командам, жюри — оценивать проекты со смартфона, а по итогам автоматически формировать дипломы для участников.

### 1.2 Целевая аудитория
- **Организаторы** — менеджеры мероприятий в образовательных учреждениях, компаниях, сообществах
- **Члены жюри** — эксперты, приглашённые для оценки проектов
- **Участники** — косвенные пользователи (получают дипломы через организатора)

### 1.3 Ключевые ценности
- Полный цикл управления мероприятием от создания до выдачи дипломов
- Мобильный интерфейс оценки для жюри — без регистрации, вход по ссылке/QR
- Работа при нестабильном интернете — оценки не теряются
- Автоматическая генерация дипломов с защитой от подделки

---

## 2. Роли пользователей

### 2.1 Организатор
- Входит в систему через логин и пароль
- Создаёт и управляет мероприятиями
- Загружает команды и участников из файлов (CSV/Excel)
- Управляет заданиями, критериями оценки, процессом выступлений
- Контролирует процесс оценки (открывает/закрывает приём оценок)
- Просматривает результаты и генерирует дипломы
- Один организатор может создавать и управлять несколькими мероприятиями

### 2.2 Жюри (Член жюри)
- Входит по уникальной ссылке или QR-коду — без регистрации
- Видит текущую выступающую команду с таймером
- Оценивает команды по заданным критериям
- Оставляет текстовые комментарии
- Видит статус своих оценок по каждой команде (не оценена / черновик / подтверждена)

### 2.3 Участник (косвенная роль)
- Не имеет доступа в систему напрямую
- Получает диплом через организатора (скачивание)
- Может проверить подлинность диплома по QR-коду

## 3. Функциональные требования

### 3.1 Управление мероприятием (Организатор)

#### FR-1: Создание мероприятия
- Организатор создаёт мероприятие с полями: название, дата проведения, описание, логотип (изображение)
- Логотип загружается как файл (JPEG, PNG), отображается на странице мероприятия и в дипломах
- Мероприятие можно редактировать после создания
- Мероприятие имеет статусы: **Черновик**, **Активно**, **Оценка завершена**, **Завершено**

#### FR-2: Критерии оценки
- Организатор задаёт список критериев оценки для мероприятия
- Каждый критерий: название, описание (необязательно), максимальный балл (целое число)
- Критерии единые для всего мероприятия (не зависят от задания)
- Можно добавлять, редактировать, удалять критерии до начала оценки
- После начала оценки изменение критериев блокируется

#### FR-3: Управление очерёдностью выступлений
- Организатор задаёт порядок выступлений команд (drag-and-drop список)
- Организатор переключает текущую выступающую команду
- Текущая команда отображается на экранах всех членов жюри в реальном времени

#### FR-4: Таймер выступления
- Организатор запускает/ставит на паузу/сбрасывает таймер обратного отсчёта
- Длительность таймера настраивается (по умолчанию — 5 минут)
- Таймер виден всем членам жюри в реальном времени
- При истечении таймера — визуальное/звуковое уведомление

#### FR-5: Управление приёмом оценок
- Организатор может открыть и закрыть приём оценок для текущей команды
- Когда приём закрыт — жюри не может выставлять/изменять оценки
- Глобальное закрытие оценки — завершение всего процесса оценки мероприятия

#### FR-6: Завершение мероприятия
- Организатор завершает мероприятие с публикацией итогов
- После завершения формируется итоговый рейтинг
- Становится доступна генерация дипломов
- Данные мероприятия сохраняются и доступны после завершения (архив)

### 3.2 Команды и участники (Организатор)

#### FR-7: Импорт команд и участников
- Поддерживаемые форматы: CSV и Excel (XLSX)
- Файл загружается через интерфейс
- После загрузки — предпросмотр данных и маппинг колонок
- Организатор указывает соответствие колонок файла полям системы:
  - Название команды (обязательное)
  - Имя участника (обязательное)
  - Email участника (необязательное)
  - Описание проекта команды (необязательное)
  - Дополнительные поля по необходимости
- Импорт создаёт команды и привязывает участников к командам

#### FR-8: Повторный импорт
- При повторном импорте система показывает предпросмотр изменений: какие команды/участники будут добавлены, обновлены или не найдут соответствия
- Организатор вручную подтверждает сопоставление импортированных строк с существующими записями (или принимает автоматическое сопоставление по названию команды)
- Уже выставленные оценки команд не теряются при обновлении данных участников
- Новые команды/участники добавляются, существующие обновляются
- Строки, которые система не может однозначно сопоставить, выделяются для ручного разрешения организатором

#### FR-9: Ручное управление
- Организатор может вручную:
  - Добавить/редактировать/удалить команду (включая название и описание проекта)
  - Добавить/редактировать/удалить участника в команде
- Описание проекта — редактируемое текстовое поле команды; может быть заполнено при импорте или вручную
- Удаление команды с оценками требует подтверждения

### 3.3 Задания (Организатор)

#### FR-10: Управление заданиями
- Организатор создаёт список заданий: название, описание, уровень сложности (низкий/средний/высокий)
- Задания можно редактировать и удалять

#### FR-11: Назначение заданий командам
- Организатор назначает задание команде вручную через интерфейс
- Возможность включить режим "одно задание — одна команда" (уникальное назначение)
- Возможность переназначить задание команде
- Одна команда имеет ровно одно назначенное задание

### 3.4 Жюри (Организатор)

#### FR-12: Управление составом жюри
- Организатор добавляет членов жюри: имя, (необязательно) email
- Для каждого члена жюри автоматически генерируется уникальная ссылка для входа
- Генерация QR-кода для каждой ссылки
- Возможность регенерировать ссылку (инвалидация старой)

#### FR-13: Мониторинг активности жюри
- Организатор видит список жюри с информацией по каждому члену:
  - Статус доступа: не заходил / заходил (с указанием времени первого входа и последней активности)
  - Текущее состояние: онлайн / офлайн
  - Прогресс оценки: количество подтверждённых оценок / общее количество команд
- Детальный просмотр: какие команды оценены каждым членом жюри, какие в черновике

### 3.5 Процесс оценки (Жюри)

#### FR-14: Вход жюри
- Жюри входит по уникальной ссылке или сканированием QR-кода
- Регистрация не требуется
- Ссылка содержит уникальный токен для идентификации

#### FR-15: Экран оценки
- Карточка текущей выступающей команды: название команды, описание проекта (из данных команды, см. FR-9), выбранное задание
- Для каждого критерия оценки — поле ввода балла с указанием максимального значения
- Отображение итоговой суммы баллов по текущей команде (автоматический подсчёт)
- Поле для текстового комментария

#### FR-16: Сохранение оценок
- Модель: **автосохранение черновика + явное подтверждение**
- По мере ввода баллов оценка автоматически сохраняется как черновик (статус "черновик")
- Член жюри нажимает кнопку "Подтвердить оценку" для финализации — после этого оценка получает статус "подтверждена"
- Подтверждённую оценку можно редактировать и повторно подтвердить, пока приём оценок открыт
- При нестабильном интернете черновики сохраняются локально и синхронизируются при восстановлении связи
- Визуальный индикатор статуса: состояние подключения, ход синхронизации, статус оценки (черновик / подтверждена)

#### FR-17: Обзор оценок жюри
- Список всех команд мероприятия с индикацией статуса: не оценена / черновик / подтверждена
- Возможность перейти к оценке конкретной команды (если приём оценок открыт)
- Возможность редактировать и повторно подтвердить ранее выставленную оценку (пока приём открыт)

### 3.6 Результаты (Организатор)

#### FR-18: Итоговая таблица результатов
- Таблица: команды (строки) x критерии (столбцы) x члены жюри
- Средний балл по каждому критерию для каждой команды
- Итоговый средний балл и рейтинг (место) команды
- Сортировка по итоговому баллу

#### FR-19: Фильтрация и анализ
- Фильтрация результатов по заданию
- Выявление аномалий: подсветка оценок, отклоняющихся от среднего более чем на заданный порог (например, > 2 стандартных отклонений)
- Возможность раскрыть детали оценки (по клику на ячейку — кто поставил, комментарий)

#### FR-20: Экспорт результатов
- Экспорт итоговой таблицы в форматах: Excel (XLSX), CSV
- Экспорт включает: команды, участников, задания, все оценки по критериям, средние баллы, места

### 3.7 Дипломы

#### FR-21: Генерация дипломов
- Автоматическая генерация PDF-диплома для каждой команды после завершения мероприятия
- Содержание диплома:
  - Логотип мероприятия
  - Название мероприятия и дата проведения
  - Место в рейтинге
  - Название команды
  - Имена участников
  - Выбранное задание
  - Итоговый балл
  - QR-код для проверки подлинности

#### FR-22: Настройка внешнего вида дипломов
- Загрузка фонового изображения для диплома
- Выбор цветовой схемы (основной цвет, цвет текста)
- Предпросмотр диплома перед массовой генерацией

#### FR-23: Верификация дипломов
- QR-код на дипломе содержит ссылку на страницу верификации
- Страница верификации доступна публично (без авторизации)
- Отображает: название мероприятия, команду, место, дату — для подтверждения подлинности

#### FR-24: Скачивание дипломов
- Скачивание диплома отдельной команды (PDF)
- Скачивание всех дипломов одним ZIP-архивом

## 4. Нефункциональные требования

### NFR-1: Mobile-first интерфейс жюри
- Интерфейс оценки полностью адаптирован под смартфон
- Touch-friendly элементы управления
- Минимальное количество скроллов для выставления оценки

### NFR-2: Offline-устойчивость
- Оценки жюри не должны теряться при кратковременной потере соединения
- При восстановлении связи данные автоматически синхронизируются с сервером
- Пользователь видит текущее состояние подключения и синхронизации
- При конфликтах (одна оценка изменена офлайн и онлайн) приоритет отдаётся последнему изменению

### NFR-3: Real-time обновления
- Текущая команда и таймер обновляются у жюри в реальном времени (задержка не более 1–2 секунд)
- Статус оценок обновляется у организатора в реальном времени

### NFR-4: Производительность
- Поддержка до 50 команд и 20 членов жюри на одно мероприятие
- Импорт файлов до 1000 строк без задержек
- Генерация дипломов — пакетная обработка

### NFR-5: Безопасность
- Токены доступа жюри должны быть уникальными, непредсказуемыми и привязанными к конкретному члену жюри
- Аутентификация организатора должна быть защищена от перебора и перехвата
- Загружаемые файлы должны проверяться на допустимый тип и размер
- Система должна быть защищена от стандартных веб-уязвимостей (инъекции, подделка запросов, XSS)
- Ссылки верификации дипломов — только для чтения, не дают доступа к управлению системой

### NFR-6: Сохранность данных
- Все данные мероприятия сохраняются после его завершения
- Завершённые мероприятия доступны в режиме "только чтение"

## 5. Пользовательские сценарии (User Stories)

### US-1: Создание и настройка мероприятия
**Как** организатор, **я хочу** создать мероприятие с критериями оценки и заданиями, **чтобы** подготовить всё к проведению.

Поток:
1. Организатор входит в систему
2. Создаёт новое мероприятие (название, дата проведения, описание, логотип)
3. Добавляет критерии оценки
4. Создаёт задания
5. Мероприятие сохраняется в статусе "Черновик"

### US-2: Импорт участников
**Как** организатор, **я хочу** импортировать команды и участников из файла Яндекс Форм, **чтобы** не вводить данные вручную.

Поток:
1. Организатор загружает CSV/XLSX файл
2. Система показывает предпросмотр данных
3. Организатор маппит колонки на поля системы
4. Система создаёт команды и участников
5. Организатор проверяет результат и при необходимости редактирует вручную

### US-3: Назначение заданий
**Как** организатор, **я хочу** назначить задания командам, **чтобы** каждая команда знала свою тему.

Поток:
1. Организатор открывает список команд
2. Для каждой команды выбирает задание из списка
3. При включённом режиме "одно задание — одна команда" уже назначенные задания недоступны для выбора

### US-4: Проведение оценки
**Как** организатор, **я хочу** управлять процессом выступлений и оценки, **чтобы** мероприятие прошло организованно.

Поток:
1. Организатор активирует мероприятие
2. Выбирает текущую выступающую команду
3. Запускает таймер
4. Открывает приём оценок
5. Жюри видят карточку команды и выставляют баллы
6. Организатор закрывает приём оценок
7. Переходит к следующей команде
8. По завершении всех выступлений — закрывает оценку мероприятия

### US-5: Оценка командой жюри
**Как** член жюри, **я хочу** оценить выступление команды со смартфона, **чтобы** быстро выставить баллы.

Поток:
1. Член жюри открывает ссылку/сканирует QR-код
2. Видит карточку текущей команды с таймером
3. Выставляет баллы по каждому критерию (черновик сохраняется автоматически)
4. Вводит комментарий (необязательно)
5. Нажимает "Подтвердить оценку" для финализации
6. Видит список всех команд и статус (черновик / подтверждена / не оценена)

### US-6: Просмотр результатов
**Как** организатор, **я хочу** увидеть итоговый рейтинг и выявить аномальные оценки, **чтобы** объективно определить победителей.

Поток:
1. Организатор открывает результаты мероприятия
2. Видит таблицу с оценками по всем командам и критериям
3. Аномальные оценки подсвечены
4. Может отфильтровать по заданию
5. Экспортирует результаты в Excel

### US-7: Генерация и скачивание дипломов
**Как** организатор, **я хочу** сгенерировать дипломы, **чтобы** каждая команда получила документ о своём участии.

Поток:
1. Организатор завершает мероприятие
2. Настраивает шаблон диплома (фон, цветовая схема)
3. Просматривает предпросмотр
4. Запускает генерацию дипломов для всех команд
5. Скачивает отдельные дипломы или весь архив

## 6. Границы проекта (Out of Scope)

Следующее **не** входит в рамки проекта:
- Самостоятельный выбор задания командой (назначает только организатор)
- Регистрация участников через систему (используются Яндекс Формы)
- Видеосвязь внутри системы
- Мобильное приложение (только мобильная веб-версия для жюри)
- Разные критерии оценки для разных заданий в рамках одного мероприятия
- Поддержка индивидуальных (не командных) соревнований
- Публичная страница мероприятия для зрителей
- Голосование зрителей / участников
- Рассылка дипломов по email
- Интеграция с внешними системами

## 7. Допущения и принятые решения

| # | Вопрос | Решение | Обоснование |
|---|--------|---------|-------------|
| 1 | Аутентификация организатора | Вход через логин и пароль | Стандартный подход для админ-панели |
| 2 | Мультитенантность | Один организатор — много мероприятий | Типичный сценарий использования |
| 3 | Формат экспорта результатов | XLSX и CSV | Наиболее универсальные форматы |
| 4 | Формат дипломов | PDF | Стандартный формат для документов |
